Backend-Refactoring: 채팅 서비스 고도화

수업 과제로 만든 채팅 프로젝트 리팩터링 기록
기능 추가보다 동시성/트랜잭션 경계/에러 응답 규격 정리에 초점

## 개요
- 기본 기능: 로그인/회원가입, 채팅방 입장/퇴장, STOMP 메시지 송수신, 메시지 히스토리 조회
- 목표: 경합/실패 상황에서도 데이터 안 깨지게 만들고, 클라에서 에러 처리할 근거(코드/규격) 남기기

## 주요 리팩터링
1. 중복 입장 및 정원 초과 처리 (완료)
- 동시 입장 시 정원 초과 가능 흐름 정리함
- `participantCount` 조건부 원자 UPDATE로 증가 처리함
- 정원 초과 시 `ROOM_FULL`로 응답하도록 보완함

2. 메시지 전송 신뢰성 정리 (진행 중)
- DB 커밋 이후(AFTER_COMMIT)에만 브로드캐스트 시도하도록 정리함
- WS 전송 실패 재시도/상태 기록은 다음 단계에서 보완 예정

3. 예외 처리 및 보안 강화 (예정)
- API 에러 응답 `ApiErrorResponse`/`ErrorCode`로 통일 정리함
- WebSocket 구독 권한은 `ChannelInterceptor`에서 참가자 검증으로 보완 예정

4. 프론트엔드 TypeScript 전환 (예정)
- 백엔드 응답/에러 규격 기준으로 TS 타입 정리 예정
- 런타임 파싱(타입가드 등) 보완 예정
